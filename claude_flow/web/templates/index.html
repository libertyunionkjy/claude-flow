Claude Flow Workflow Guide

Claude Flow uses a task queue + Git Worktree isolation model for multi-instance parallel development.
Below is the complete lifecycle of a task -- click each step to expand details.

1 Initialize Project Setup ▼

First, initialize Claude Flow in your project root. This creates a .claude-flow/ directory containing configuration and task data.

CLIcf init
WebAuto-initialized when Web Manager starts
Tip: Run cf init once per project. Works with both Git repos and non-Git directories (non-Git mode uses single worker without worktree isolation).
2 Add Tasks Pending ▼

Create tasks with a title and a prompt. The prompt is what Claude Code will receive as instructions. Choose between two task types:

  • Normal Task - Full lifecycle with planning, review, and execution. Best for complex features.
  • Mini Task - Skips planning/approval, goes directly to Approved. Ideal for quick fixes and simple changes.
CLIcf task add -p "prompt" "title"
cf task mini "prompt" --run
WebClick + New Task, select Normal or Mini type
Tip: Use -f tasks.txt to batch-add from a file. Set priority with -P flag. Mini tasks with --run execute immediately.
3 Plan Tasks Planning AUTO ▼

Choose a planning mode for each task (mini tasks skip this step entirely):

  • Auto Plan - AI generates a complete plan in the background (best for bug fixes and small features)
  • Chat Plan - Multi-round conversation to refine requirements and design before generating the plan
CLIcf plan (background) or cf plan -F (foreground)
cf plan -t <id> --interactive (chat mode)
cf plan status (check progress)
WebClick Auto Plan or Chat Plan on the card

Plans are generated in the background by default. Use cf plan status to check progress. Both modes produce a Markdown plan document in .claude-flow/plans/.

4 Review & Approve Plans Planned HUMAN ▼

This is the human-in-the-loop gate. Review each plan to ensure correctness and safety before approving execution. Mini tasks skip this step.

CLIcf plan review (interactive)
cf plan approve --all (quick approve)
WebClick View Plan to read, then Approve or Chat
Chat: Open the chat dialog to discuss the plan with AI, ask questions, or request revisions. Click Finalize Plan in the chat to regenerate the plan from the conversation history.
5 Execute Tasks Approved AUTO ▼

Workers pick up approved tasks (both normal and mini) and execute them. Each task gets its own Git Worktree (isolated branch cf/task-XXXXXX) so multiple tasks can run in parallel without conflicts.

CLIcf run (single) or cf run -n 3 (parallel)
cf run --daemon (continuous polling)
WebClick Run All in toolbar or Run on individual cards
Daemon: Use cf run --daemon for continuous execution. Workers poll for new tasks every 10 seconds (configurable via daemon_poll_interval). Stop with Ctrl+C.
6 Running & Needs Input Running ▼

While running, Claude Code writes code in the isolated worktree. Monitor real-time progress via structured logs that show tool calls, errors, and execution timeline.

If Claude Code produces no code changes (e.g., needs clarification), the task enters Needs Input status with Claude's response saved. You can provide additional context and the task will be re-queued.

CLIcf log <task_id> (structured log)
cf respond <task_id> -m "info" (answer Claude)
cf watch (live monitor)
WebView Log on running cards. Respond button for Needs Input cards.
Safety: Workers detect if Claude accidentally modifies the main repo and automatically rescue changes to the worktree.
7 Test, Merge & Done Done AUTO ▼

After execution, the worker performs a multi-stage merge pipeline:

  • Auto-commit any uncommitted changes in the worktree
  • Pre-merge tests (if configured) -- run test commands, auto-fix failures with Claude
  • Rebase merge to main branch with merge lock (prevents concurrent merges)
  • Conflict auto-resolution -- Claude resolves conflicts with task context awareness
  • Remote push (if auto_push enabled)
  • Experience log to PROGRESS.md
CLIcf clean (cleanup)
cf progress (experience log)
cf usage (token stats)
WebTask card moves to Done automatically. View usage stats in the Overview dashboard.
Config: Set auto_merge: false to skip merging. Use pre_merge_commands: ["pytest -v"] to enable test verification. Merge strategy defaults to rebase with max_merge_retries: 5.
8 Handle Failures Failed HUMAN ▼

If a task fails (timeout, Claude error, merge conflict, test failure), it enters Failed status with an error message. You can:

  • Retry -- re-queue as Approved (keeps the plan)
  • Reset -- send back to Pending (mini tasks reset to Approved)
  • View Log -- inspect structured execution log with tool calls and errors
  • Delete -- remove the task (auto-stops running processes and cleans worktree)
CLIcf retry (all failed)
cf reset <task_id> (single task)
cf log <task_id> (inspect)
WebUse Retry, Reset, or Delete buttons on failed cards. Batch delete via task selection.
Zombie tasks: If a worker crashed, use cf reset <task_id> to reset zombie running tasks. This also cleans up orphaned worktrees and branches.
9 Token Usage & Monitoring Stats ▼

Track token consumption across all tasks and sessions. Usage data is collected from stream-json logs, with enhanced reporting available via ccusage.

CLIcf usage (per-session report)
cf usage daily / monthly / summary
cf usage --since 2026-03-01
WebOverview dashboard shows aggregated usage stats. API endpoints available at /api/usage/*.
Tip: Install ccusage (npx ccusage@latest) for full-featured daily/monthly reporting. Without it, basic per-task stats are still available from execution logs.
